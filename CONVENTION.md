# 팀 컨벤션

## 브랜치 전략

1. 각자의 브랜치에서 작업 후 `main` 브랜치로 Pull Request 전송
2. **1명 이상의 BE 개발자 코드 리뷰** 이후 main 브랜치로 머지 가능

<br>

## 커밋 메시지

```
type(scope): subject

ex) feat(auth): 카카오 로그인 구현
```

- 제목과 본문은 **빈 행으로 구분**
- 어떻게 보다는 **무엇과 왜**를 설명
- 제목은 **명령형 동사**, 과거형 사용 금지, 마침표 없음

| 타입 | 의미 |
|---|---|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 코드 리팩토링 (기능 변화 없이 구조 개선) |
| `chore` | 자잘한 수정 |
| `build` | 빌드 관련 파일 수정 |
| `docs` | 문서 수정 |
| `test` | 테스트 코드 추가 또는 수정 |
| `deploy` | 인프라, 배포, CI/CD 관련 수정 |

<br>

## 코드 컨벤션

- **단일 책임 원칙(SRP)** 준수
- 메서드 길이는 **30줄 이하**, 내부 중첩은 **2depth까지**
- 생성자를 이용해 명시적으로 의존성 주입
- `else` 사용 지양
- 새줄 문자는 `LF`
- Request/Response DTO는 `record` 클래스 사용

<br>

## 메서드 네이밍

| 동작 | prefix |
|---|---|
| 생성 | `create` |
| 수정 | `update` |
| 삭제 | `delete` |
| 조회 | `find` |
